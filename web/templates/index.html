RPM Translations subpackaging

Problem Statement

Size

Currently most of the RPM packaged applications ship the translations with them. Translations take up more space (even than fonts) in fedora workstation.

Hence there could be a need to limit this considering fedora minimization initiative.

Expected Benefits

  • If we only install translation langpacks for the system/user locale that will reduce the footprint a lot. Moreover, this will help with the minimization efforts.
  • This could also be used if we will create the locale extensions for fedora flatpaks.

References

Team JIRA | Silverblue Ticket | RPM Upstream

Investigate

  • number of packages affected - see reports
  • number of .mo files and their size - see reports (for fedora base img): silverblue | workstation | server
  • which binary package the translations live in now - see reports

Probable Solution/Options

What to solve

  1. how to generate names for subpackages, on which types/names of files, which Provides/Supplements, how they would be tied to the original package where translations are coming from. In other words, what MAINPKG would be (how to automatically determine it) and what XX to use?
  2. the support of automated subpackage generation from RPM, that is something what needs to be implemented in RPM.
